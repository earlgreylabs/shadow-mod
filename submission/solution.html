How ShadowMod Works
1

Shadow mod records a decision

A new mod sees a post in the queue and selects "Record shadow decision" from the post menu. They choose an action and write their reasoning. The action is not executed — nothing happens to the post.

Menu action → Form → Redis
2

Senior mod reviews independently

A senior mod sees "Review shadow decision" on the same post and records their own call — without seeing the trainee's decision first. This blind review eliminates anchoring bias: the feedback is genuinely independent.

Blind review → Form → Redis
3

Real action taken in normal workflow

The senior mod (or any other mod) takes the real action on the post through the normal Reddit mod queue. ShadowMod does not interrupt or replace this step.

Normal Reddit workflow
4

Trigger detects the action

ShadowMod's onModAction trigger fires automatically. It detects that this post had a pending shadow+senior decision pair and schedules a report job.

Devvit trigger → Scheduler
5

Comparison report delivered

The trainee receives a structured report via modmail: their decision vs the senior's decision vs the final outcome — with reasoning from both sides. A mod note is written as fallback if modmail fails.

Modmail → Mod note fallback
6

Stats update

The trainee's accuracy stats in Redis are updated. Over time, longitudinal patterns emerge: which content types they get right, where they diverge from senior judgment, and how they improve.

Redis stats

Why it matters

Most training mistakes happen in the grey areas — whether to warn vs. ban, flair vs. remove. Binary approve/remove training doesn't prepare mods for these decisions.

Key Design Principles
Blind senior review

Senior records their call before seeing the trainee's. Without this, feedback becomes rationalisation of what the trainee said — not genuine independent judgment.

Async, not immediate

Feedback arrives after the senior reviews — not instantly. This mirrors real moderation flow and prevents trainees from gaming the system for validation.

No workflow disruption

ShadowMod listens passively via the onModAction trigger. Senior mods take real actions exactly as before — nothing changes for them until they opt in to review.

Community-configurable

Senior mods are defined per subreddit in app settings. Each community controls who reviews whom.

Technical Implementation
Platform Devvit (Reddit Developer Platform) · @devvit/web v0.12
Server Hono + Node · TypeScript · routes for menu, forms, triggers, scheduler
Storage Devvit Redis · namespaced per subreddit installation · hashes + sorted sets
Key data decision:{postId}:{modId} · senior:{postId}:{modId} · pending (sorted set) · stats:{userId}
Role gating forUserType: "moderator" in devvit.json · shadow vs senior tracked in config:seniorMods
Report trigger onModAction trigger → detects action on posts with pending decision pairs → scheduler.runJob() → modmail delivery
Install One-click install from Reddit developer marketplace · configure senior mods via subreddit menu action

Community Impact

Reduces bad early-mod decisions before they reach community members. Scales senior mod mentorship without adding hours. Measurable: accuracy % per trainee over time.

Reliable UX

One-click install. No configuration required to start — shadow decisions work immediately. Senior mods added via a single subreddit menu action. No external dependencies.

Ecosystem Impact

Applicable to any subreddit with a tiered mod structure. As subreddits grow and recruit new mods, ShadowMod provides a repeatable onboarding system — not a one-time fix.
